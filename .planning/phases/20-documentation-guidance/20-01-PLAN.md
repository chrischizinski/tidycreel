**Structure:**

1. **Introduction** (why this vignette exists)
   - Roving-access design uses complete trips by default (Pollock et al.)
   - Incomplete trips CAN be valid with proper validation
   - This vignette shows when and how to use incomplete trip estimation

2. **Scientific Rationale** (DOC-01)
   - Explain roving-access design theory (counts + complete trips)
   - Why complete trips are preferred (no length-of-stay bias)
   - When incomplete trips might be considered (large sample, validation passes)
   - Cite Pollock et al., Hoenig et al., Colorado C-SAP best practices
   - Explain mean-of-ratios estimator vs ratio-of-means

3. **Colorado C-SAP Best Practices** (DOC-02)
   - Default to complete trips only (package default behavior)
   - Require ≥10% complete trip interviews
   - Only use incomplete trips after validation
   - Example showing complete trip workflow (reference interview-estimation.Rmd)

4. **Strong Warning: Never Pool Complete + Incomplete** (DOC-03)
   - Create callout box with cli formatting
   - Explain why pooling is scientifically invalid (different sampling probabilities)
   - Show what NOT to do (anti-pattern example)
   - Emphasize package prevents auto-pooling by design

5. **When to Consider Incomplete Trips**
   - Scenarios: low complete trip sample size, research questions about incomplete trips
   - Prerequisites: sufficient incomplete sample (n≥30), similar fishing behavior
   - Validation REQUIRED before using incomplete trip estimates

6. **Step-by-Step Validation Workflow** (DOC-04)
   - Load data with both complete and incomplete trips
   - Step 1: Check sample sizes (≥10% complete per Colorado C-SAP)
   - Step 2: Run validate_incomplete_trips() with TOST equivalence testing
   - Step 3: Interpret results (overall + per-group if grouped)
   - Step 4: View validation plot
   - Step 5: Make decision based on validation outcome

7. **Example: Validation Passes** (realistic scenario)
   - Simulate data where catch rates are similar for complete vs incomplete
   - Show full workflow: design setup, add_interviews, validate_incomplete_trips
   - Interpret passing TOST results
   - Show it's acceptable to use use_trips="incomplete" or "diagnostic" mode
   - Show print output and plot

8. **Example: Validation Fails** (realistic scenario)
   - Simulate data where incomplete trips have bias (e.g., early anglers catch more)
   - Show same workflow but validation fails
   - Interpret failing TOST results
   - Recommendation: stick with complete trips only
   - Show print output and plot demonstrating bias

9. **Using Diagnostic Comparison Mode**
   - Show use_trips="diagnostic" for side-by-side comparison
   - Interpret difference metrics (10% threshold guidance from Phase 17)
   - When diagnostic mode is useful vs validate_incomplete_trips()

10. **Advanced: Grouped Validation**
    - Example with by= parameter (e.g., by day_type)
    - Conservative approach: overall AND all groups must pass
    - Interpret per-group validation results

11. **Technical Details**
    - TOST equivalence testing explained (refer to ?validate_incomplete_trips)
    - Equivalence threshold configuration (tidycreel.equivalence_threshold option)
    - Trip truncation (truncate_at parameter, default 30 min)
    - Mean-of-ratios variance estimation

12. **Summary and Recommendations**
    - Flowchart or decision tree: Should I use incomplete trips?
    - Default recommendation: complete trips only
    - If considering incomplete: validate first
    - Never pool complete + incomplete
