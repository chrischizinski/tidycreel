**Audit checklist (D-14):**

(a) No plaintext credential logging: scan for any call to `message()`, `cat()`, `print()`, or `cli::cli_inform()` that references `auth$token`, `auth$key`, or any variable holding a password. Zero hits required.

(b) No un-sanitized user input in URL/SQL strings: verify URL query construction uses `httr2::req_url_query()` (already present at creel-connection-api.R lines ~179–181) — not paste0/sprintf with creel_uids interpolated directly. There is no SQL code in tidycreel.connect, so SQL injection does not apply.

(c) Credentials sourced only from env vars or YAML `!expr Sys.getenv()`: scan for string literals that look like tokens/passwords (any `= "..."` assignment to a field named `token`, `key`, `password`, `username`). Zero hard-coded credential values allowed.

(d) `auth$token` not printed, serialized, or stored beyond request scope: verify no code path stores `auth$token` into a persistent object, file, or log stream.

**Protective comment to add (regardless of audit result):**

In `creel-connection-api.R`, locate the bearer token injection block (around lines 185–188):
```r
if (auth$type == "bearer") {
  req <- httr2::req_auth_bearer_token(req, auth$token)
}
```

Add this comment immediately before the `if` block:
```r
# WARNING: Do NOT print auth or req objects after this point — auth$token will
# leak to logs. Use httr2::req_dry_run() for debugging; httr2 redacts auth headers.
```

**Fix protocol:**
- If a violation is found in any of the four files, fix it in place (e.g., remove a `message()` call, replace a hard-coded string with `Sys.getenv()`).
- If no violations are found (expected outcome based on research), the only change is the protective comment in `creel-connection-api.R`.
- Do NOT restructure code, rename functions, or change logic beyond the specific violation fix.
