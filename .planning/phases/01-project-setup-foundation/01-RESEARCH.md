## Summary

This phase initializes a clean R package structure from scratch, replacing the existing v1 codebase with an empty v2 scaffold. The standard approach uses `usethis` helpers for package scaffolding, `r-lib/actions` for GitHub Actions CI/CD, `lorenzwalthert/precommit` for pre-commit hooks with lintr/styler enforcement, testthat 3rd edition for testing, and `checkmate` for data schema validation.

The key complexity in this phase is that the project already has a v1 codebase in the working directory. The v2 build starts from an empty package on a new branch (`v2-foundation`), so all v1 R source files, tests, and NAMESPACE exports must be removed and replaced with a minimal scaffold. The existing `.github/workflows/`, `.pre-commit-config.yaml`, and `.lintr` files serve as useful references but need significant updates to match v2 requirements.

The CONTEXT.md decisions constrain this work clearly: use usethis for scaffolding, MIT license, tidyverse lintr defaults, standard tidyverse styler, checkmate for validation, README.Rmd in Phase 1, latest R release only for CI (not devel/oldrel), but full cross-platform (Linux + Windows + macOS).

**Primary recommendation:** Use `usethis` to scaffold a fresh package, configure `r-lib/actions@v2` workflows with `check-r-package@v2` for CI, set up `lorenzwalthert/precommit` with lintr + style-files hooks, and implement schema validators using `checkmate` assertions with cli error messages.

### Pattern 1: Clean Slate Package Scaffolding

**What:** Remove all v1 R source, tests, NAMESPACE exports, and v1-specific DESCRIPTION dependencies. Rebuild using `usethis` helpers.
**When to use:** Starting v2 from scratch on a new branch.
**Why:** The existing repo has ~40 R files, 20+ test files, and a NAMESPACE with 97 exports. Starting clean avoids inheriting v1 technical debt.

**Approach:**
1. Remove all files in `R/`, `tests/testthat/`, `man/`, `vignettes/`
2. Regenerate NAMESPACE via roxygen2 (empty or minimal)
3. Rewrite DESCRIPTION with v2 minimal dependencies
4. Use `usethis::use_testthat(3)` to set up test infrastructure
5. Use `usethis::use_package_doc()` for package-level documentation

### Pattern 2: Schema Validation with checkmate + cli

**What:** Define data schema validators as functions that use checkmate assertions internally, with cli for user-friendly error messages.
**When to use:** Validating user-provided data frames (calendar, counts) before processing.

**Key design decisions:**
- Use `checkmate::assert_data_frame()` for structural checks
- Use `checkmate::assert_names()` with `must.include` for required columns
- Use individual column type assertions (`assert_date()`, `assert_integerish()`, etc.)
- Wrap in `tryCatch` or use `checkmate::makeAssertCollection()` to collect all errors before reporting
- Use `cli::cli_abort()` with informative messages that explain both what failed AND how to fix

### Pattern 3: GitHub Actions with r-lib/actions@v2

**What:** Use the standard r-lib/actions workflow templates for R CMD check and coverage.
**When to use:** All R package CI/CD.

**Key configuration per CONTEXT.md decisions:**
- Latest R release only (no devel, no oldrel) -- simplifies matrix
- Full cross-platform: ubuntu-latest, macos-latest, windows-latest
- Use `r-lib/actions/check-r-package@v2` (the modern replacement for manual rcmdcheck calls)
- Aggressive caching via `setup-r-dependencies@v2` (enabled by default)

### Anti-Patterns to Avoid

- **Keeping v1 code "just in case":** The v1 code is preserved in the `main` branch and `.planning-legacy-work/`. Do NOT leave v1 files in the v2 branch.
- **Hand-writing NAMESPACE:** Always use roxygen2-generated NAMESPACE. Never manually edit.
- **Installing from source in CI:** Use `setup-r-dependencies@v2` which handles binary packages and caching automatically.
- **Skipping R CMD check locally:** Run `devtools::check()` before pushing. CI catches what local misses, but don't rely solely on CI.
- **Complex .lintr customizations:** Start with `linters_with_defaults()` and only override line length. Don't disable linters without good reason.

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Package scaffolding | Manual file creation | `usethis::create_package()` / helpers | Gets DESCRIPTION format, NAMESPACE, directory structure, .Rbuildignore all correct |
| GitHub Actions workflow | Write YAML from scratch | `usethis::use_github_action("check-standard")` then customize | r-lib maintains these; handles OS-specific quirks (macOS Homebrew, Windows Rtools) |
| Pre-commit hook config | Custom git hooks | `precommit::use_precommit()` | Cross-platform, maintained, handles R environment setup in hooks |
| Data frame validation | Manual `if/else` chains | `checkmate::assert_data_frame()` + `assert_names()` | Handles edge cases (NULL, empty, wrong class), C-optimized, clear error messages |
| Test infrastructure | Manual test directory setup | `usethis::use_testthat(3)` | Sets Config/testthat/edition, creates directory structure, writes test runner |
| License files | Manual LICENSE creation | `usethis::use_mit_license()` | Creates both LICENSE and LICENSE.md, updates DESCRIPTION |
| README template | Manual README | `usethis::use_readme_rmd()` | Creates README.Rmd with proper .Rbuildignore entry |
| Coverage setup | Manual covr config | `usethis::use_coverage("codecov")` | Adds covr to Suggests, creates badge, notes about CODECOV_TOKEN |

**Key insight:** The R ecosystem has mature tooling for package setup. Nearly every scaffolding task has an `usethis::use_*()` function. Using these ensures standard conventions and avoids subtle mistakes (e.g., wrong .Rbuildignore regex, missing Config/ fields in DESCRIPTION).

## Common Pitfalls

### Pitfall 1: v1 Artifacts Causing R CMD Check Failures

**What goes wrong:** Leftover v1 files (R source with undocumented exports, test files referencing missing functions, man pages for removed functions) cause R CMD check errors/warnings/notes.
**Why it happens:** The v2 branch inherits the full v1 file tree. Incomplete cleanup leaves orphaned files.
**How to avoid:** Systematically delete ALL v1 content from `R/`, `tests/testthat/`, `man/`, `vignettes/` before scaffolding v2. Verify with a fresh `devtools::check()`.
**Warning signs:** R CMD check shows "Undocumented objects", "Objects in \\usage without \\alias", or "Namespace dependency not required" notes.

### Pitfall 2: DESCRIPTION Dependencies Mismatch

**What goes wrong:** DESCRIPTION lists packages that are not actually used (v1 leftovers like lme4, plotly, ggplot2), or missing packages that are used.
**Why it happens:** v1 DESCRIPTION has 10+ Imports and 15+ Suggests. v2 starts with minimal dependencies.
**How to avoid:** Write a fresh DESCRIPTION with ONLY the packages needed for Phase 1. For Phase 1, Imports should be: cli, rlang, checkmate. Everything else is Suggests or deferred to later phases.
**Warning signs:** R CMD check shows "Namespace dependency not required" or "All declared Imports should be used".

### Pitfall 3: Pre-commit Framework Not Installed

**What goes wrong:** `precommit::use_precommit()` fails because the Python `pre-commit` framework is not installed on the system.
**Why it happens:** The R `precommit` package is just an interface; the actual hook framework is a Python tool that must be installed separately.
**How to avoid:** Install pre-commit framework first: `pip3 install pre-commit --user` or `brew install pre-commit` on macOS. Then install the R package. Verify with `pre-commit --version` at the command line.
**Warning signs:** Error about "pre-commit executable not found" or hooks not running on commit.

### Pitfall 4: "No Visible Binding for Global Variable" Notes

**What goes wrong:** R CMD check produces notes about unbound global variables when using tidy evaluation (bare column names in dplyr pipelines).
**Why it happens:** R's static analysis cannot resolve non-standard evaluation patterns.
**How to avoid:** Use `utils::globalVariables()` in a `globals.R` file, OR use `.data$column_name` pronoun from rlang. The `.data` pronoun is preferred in modern tidyverse packages.
**Warning signs:** NOTE: "no visible binding for global variable 'column_name'"

### Pitfall 5: R CMD Check Notes on New Submission

**What goes wrong:** R CMD check --as-cran produces a NOTE about "New submission" or "Checking CRAN incoming feasibility".
**Why it happens:** This is unavoidable for packages not yet on CRAN.
**How to avoid:** For now, use `--no-manual` and avoid `--as-cran` in CI if this note is treated as failure. The existing v1 workflow already uses `--no-manual`. Consider using `r-lib/actions/check-r-package@v2` which handles this gracefully.
**Warning signs:** CI fails on a NOTE that cannot be fixed.

### Pitfall 6: Coverage Token Missing

**What goes wrong:** Test coverage workflow runs successfully but coverage is not uploaded to Codecov.
**Why it happens:** Codecov now requires a `CODECOV_TOKEN` secret for reliable uploads, even for public repos.
**How to avoid:** Set up Codecov account, get upload token, add as GitHub repository secret named `CODECOV_TOKEN`.
**Warning signs:** Coverage badge shows "unknown" or gray.

### Pitfall 7: Pre-commit Hooks Too Strict Initially

**What goes wrong:** Developers cannot commit because lintr finds issues in files they did not change, or styler reformats everything.
**Why it happens:** Running lintr/styler on the entire repo catches v1 legacy issues.
**How to avoid:** Since v2 starts from scratch, this is mainly a risk if any v1 files remain. After cleanup, all new code should pass from the start. Use `--warn_only` flag on lintr hook during initial setup if needed, then switch to strict mode.
**Warning signs:** Every commit attempt blocked by lint errors in files not touched.

### Schema Validator Pattern with checkmate

```r
# Source: https://mllg.github.io/checkmate/reference/checkDataFrame.html
# Pattern for validate_calendar() in v2

#' Validate Calendar Data Schema
#'
#' Validates that calendar data conforms to the expected schema for
#' creel survey design construction.
#'
#' @param data A data frame containing calendar data.
#' @return Invisibly returns the validated data frame.
#' @keywords internal
validate_calendar_schema <- function(data) {
  # Collect all errors before reporting
  collection <- checkmate::makeAssertCollection()

  # Structural check
  checkmate::assert_data_frame(data, min.rows = 1, add = collection)

  # Required columns
  required_cols <- c("date", "day_type")
  checkmate::assert_names(
    names(data),
    must.include = required_cols,
    add = collection
  )

  # Type checks (only if columns exist)
  if ("date" %in% names(data)) {
    checkmate::assert_date(data$date, any.missing = FALSE, add = collection)
  }

  if ("day_type" %in% names(data)) {
    checkmate::assert_character(
      data$day_type,
      any.missing = FALSE,
      add = collection
    )
  }

  # Report all collected errors at once
  if (!collection$isEmpty()) {
    msgs <- collection$getMessages()
    cli::cli_abort(c(
      "Calendar data validation failed:",
      set_names(msgs, rep("x", length(msgs))),
      "i" = "See {.fun creel_design} for expected calendar format."
    ))
  }

  invisible(data)
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `rcmdcheck::rcmdcheck()` in CI steps | `r-lib/actions/check-r-package@v2` | r-lib/actions v2 (2022+) | Handles check, upload snapshots, build args in one action step |
| `covr::codecov()` direct upload | `covr::to_cobertura()` + `codecov/codecov-action@v5` | codecov-action v4/v5 (2024+) | Token required; cobertura XML format; more reliable uploads |
| testthat 2e with `context()` | testthat 3e, file-based contexts | testthat 3.0.0 (2020) | `context()` removed; `expect_is()` deprecated; waldo-based equality |
| Manual `.lintr` with individual linters | `linters_with_defaults()` | lintr 3.0.0 (2022) | Declarative config; override only what differs from tidyverse defaults |
| Custom git hook scripts | `lorenzwalthert/precommit` R package | precommit 0.1.0 (2020) | Cross-platform, maintained, version-pinned hooks |
| Manual `setup-r`, `install.packages()` in CI | `setup-r-dependencies@v2` with pak | r-lib/actions v2 (2022) | Binary packages, lock files, automatic system deps on Linux |

**Deprecated/outdated:**
- `testthat::context()`: Removed in 3e; use file names for context
- `testthat::expect_is()`: Replaced by `expect_s3_class()`, `expect_type()`
- `testthat::expect_equivalent()`: Use `expect_equal(ignore_attr = TRUE)`
- `covr::codecov()` direct upload without token: Codecov requires CODECOV_TOKEN as of 2024
- `r-lib/actions@v1`: Use v2 for all actions

## Open Questions

Things that couldn't be fully resolved:

1. **Exact v2 calendar schema columns**
   - What we know: v1 had 10 required columns (date, stratum_id, day_type, season, month, weekend, holiday, shift_block, target_sample, actual_sample). v2 design document says calendar needs "dates, strata" at minimum, with tidy selectors specifying which columns serve which roles.
   - What's unclear: The exact minimal required columns for v2 calendar schema. Since `creel_design()` uses tidy selectors, the schema might only need to validate that the user-specified columns exist and have correct types, rather than requiring fixed column names.
   - Recommendation: For Phase 1, define minimal structural validators (data frame with at least one Date column and character columns for strata). Defer specific column validation to Phase 2 when `creel_design()` is built and column roles are known. Phase 1 schemas should validate structure and types, not specific column names.

2. **Exact v2 count schema columns**
   - What we know: v1 had 12 required columns. v2 uses tidy selectors so count columns are user-specified.
   - What's unclear: Same as calendar -- which columns are structurally required vs. user-specified via tidy selectors.
   - Recommendation: Same approach. Validate that the data is a data frame with reasonable structure (has rows, has columns with expected types). Specific column role validation happens in Phase 3 (`add_counts()`).

3. **Pre-commit blocking vs. warning behavior**
   - What we know: CONTEXT.md lists this as Claude's Discretion. lintr hook supports `--warn_only` argument.
   - What's unclear: Whether strict blocking is desirable from day one.
   - Recommendation: Use strict blocking (no `--warn_only`) since v2 starts from scratch with clean code. Style-files hook should auto-fix. Lintr hook should block. This matches the project constraint "All code passes lintr before commit."

4. **Coverage threshold enforcement**
   - What we know: CONTEXT.md lists this as Claude's Discretion. Project requires 85% overall, 95% for core estimation.
   - What's unclear: Whether to enforce thresholds in CI in Phase 1 (when there's almost no code to cover).
   - Recommendation: Track coverage but do NOT enforce thresholds in Phase 1. Add threshold enforcement in Phase 7 when the codebase is complete. Configure Codecov to report but not fail.

5. **R CMD check note handling**
   - What we know: CONTEXT.md lists this as Claude's Discretion. Success criterion says "no errors/warnings/notes".
   - What's unclear: Some notes are unavoidable (e.g., "New submission" on --as-cran).
   - Recommendation: Use `r-lib/actions/check-r-package@v2` default behavior which fails on warnings and errors but not notes. The success criterion "no notes" can be verified manually. Don't use `--as-cran` flag which adds CRAN-specific notes.
